### 1.2 各貼文類型的問題分析

#### 故事型（story）

| 問題 | 說明 | 影響 |
|------|------|------|
| **輸入欄位過於抽象** | `event_conflict`、`turning_point` 等欄位的 placeholder 不夠具體 | 用戶輸入抽象內容，AI 被迫編造細節 |
| **強制英雄旅程架構** | 所有故事都套用「衝突→轉折→啟發」結構 | 故事變得公式化 |
| **禁止虛構場景** | 規則說「禁止虛構」，但沒有引導用戶提供真實細節 | 矛盾：AI 不能編，用戶也沒給 |

**範例問題**：
- 用戶輸入：「很多老師擔心漲價」
- AI 被迫編造：「昨天有個學員跟我說...」（違反禁止虛構規則）
- 或者：AI 寫出抽象的觀點文（不符合故事型特徵）

#### 觀點型（viewpoint）

| 問題 | 說明 | 影響 |
|------|------|------|
| **立場不夠鮮明** | 用戶常輸入「我覺得漲價是好的」這種弱立場 | 生成的內容沒有張力 |
| **缺少論點引導** | 沒有引導用戶提供 2-3 個支撐論點 | AI 自己編論點，變得空洞 |

#### 對話型（dialogue）

| 問題 | 說明 | 影響 |
|------|------|------|
| **對話角色不明確** | `dialogue_roles` 欄位太籠統 | AI 編造不自然的對話 |
| **缺少真實對話內容** | 沒有引導用戶輸入「原話」 | AI 編造的對話很 AI |

#### 知識型（knowledge）

| 問題 | 說明 | 影響 |
|------|------|------|
| **專業術語翻譯不足** | `professional_concept` 欄位常被跳過 | 內容變成說教 |
| **步驟過於抽象** | `key_points` 常被輸入為「三個重點」 | AI 自己編重點 |

#### 整理型（summary）

| 問題 | 說明 | 影響 |
|------|------|------|
| **原始資料不足** | `raw_data` 欄位常被跳過 | AI 自己編清單 |
| **數字不具體** | 沒有引導用戶決定「幾個」 | AI 隨意決定數量 |

#### 診斷型（diagnosis）

| 問題 | 說明 | 影響 |
|------|------|------|
| **症狀不夠具體** | `symptoms` 常被輸入為抽象描述 | AI 編造不準確的症狀 |
| **標籤不夠有趣** | `diagnosis_label` 常被跳過 | 失去診斷型的趣味性 |

#### 閒聊型（casual）

| 問題 | 說明 | 影響 |
|------|------|------|
| **生活碎片不具體** | `life_fragment` 常被跳過 | AI 編造不真實的場景 |
| **情緒詞過度使用** | 系統鼓勵用語氣詞，但用太多反而 AI 感 | 「超累啊～真的欸～」 |

#### 提問型（question）

| 問題 | 說明 | 影響 |
|------|------|------|
| **問題太複雜** | 沒有引導用戶簡化問題 | 生成的問題太長，沒人想回答 |

#### 投票型（poll）

| 問題 | 說明 | 影響 |
|------|------|------|
| **選項不夠對立** | `binary_choice` 常被輸入為相似選項 | 投票沒有張力 |

#### 引用型（quote）

| 問題 | 說明 | 影響 |
|------|------|------|
| **延伸觀點不足** | `extended_view` 常被跳過 | 變成純引用，沒有個人觀點 |

#### 反差型（contrast）

| 問題 | 說明 | 影響 |
|------|------|------|
| **反差不夠大** | `two_opposites` 常被輸入為弱對比 | 沒有驚喜感 |

### 1.3 AI 感的具體來源

經過 A/B 測試和分析，AI 感主要來自以下原因：

| 來源 | 具體表現 | 根本原因 |
|------|----------|----------|
| **開頭公式化** | 每篇都是「冠號斷言」格式 | 系統強制使用數據驅動開頭 |
| **結尾公式化** | 每篇都是「你們覺得呢？」 | CTA 規則太單一 |
| **語氣詞過度** | 「親愛的」「真的」「超」出現太多 | IP 地基的情感共鳴被過度套用 |
| **結構太工整** | 每段都是 2-4 行，太規律 | 呼吸感排版規則太嚴格 |
| **缺少具體細節** | 沒有具體數字、時間、人物 | 用戶輸入太抽象，AI 被迫編造 |
| **Few-Shot 複製** | 開頭句式跟範文一樣 | Few-Shot 學到句式而非風格 |

## 二、爆款選題庫功能設計

### 2.1 功能定位

**核心問題**：創作者常常「不知道要寫什麼」，即使有 IP 地基和受眾資料，也不知道如何轉化成具體的貼文主題。

**解決方案**：利用現有的 1,739+ 篇爆款資料庫，建立「爆款選題庫」功能，幫助創作者找到「有話想說」的選題。

### 2.3 技術實作方案

#### 資料來源

利用現有的爆款資料庫：
- `viral_posts` 表：1,739+ 篇爆款貼文
- `content_clusters` 表：主題群集分析
- `topic_suggestions` 表：選題建議
- `success_factors` 表：成功因素分析

#### API 設計

```typescript
// 1. 取得推薦選題（情境 A：完全沒想法）
getRecommendedTopics: protectedProcedure
  .input(z.object({
    count: z.number().default(5),
  }))
  .query(async ({ ctx, input }) => {
    // 根據用戶的領域和受眾，推薦相關選題
    const profile = await db.getIpProfileByUserId(ctx.user.id);
    const audiences = await db.getAudienceSegmentsByUserId(ctx.user.id);
    
    // 從 topic_suggestions 和 content_clusters 中篩選
    const topics = await db.getTopicsByDomain(profile?.domain, input.count);
    
    return topics.map(t => ({
      id: t.id,
      title: t.template,
      viralRate: t.viralRate,
      recommendedTypes: t.recommendedTypes,
      samplePost: t.samplePost,
    }));
  }),

// 2. 搜尋相關切角（情境 B：有點想法）
searchTopicAngles: protectedProcedure
  .input(z.object({
    keyword: z.string(),
    count: z.number().default(5),
  }))
  .query(async ({ ctx, input }) => {
    // 用 Embedding 搜尋相關的爆款貼文
    const similarPosts = await findSimilarViralExamples(input.keyword, input.count * 2);
    
    // 分析這些爆款的切角
    const angles = analyzeAngles(similarPosts);
    
    return angles.slice(0, input.count).map(a => ({
      angle: a.angle,
      viralRate: a.viralRate,
      recommendedType: a.recommendedType,
      coreArgument: a.coreArgument,
      samplePosts: a.samplePosts.slice(0, 2),
    }));
  }),

// 3. 取得具體化引導問題
getGuidingQuestions: protectedProcedure
  .input(z.object({
    topicId: z.number().optional(),
    angle: z.string().optional(),
    contentType: z.string(),
  }))
  .query(async ({ ctx, input }) => {
    // 根據選題和內容類型，生成引導問題
    const questions = generateGuidingQuestions(input.contentType, input.angle);
    
    return questions;
  }),
```

## 三、A/B 測試方案

### 3.1 測試設計

為了驗證優化效果，我將使用同一套輸入內容，針對不同類型的貼文進行測試，並呈現提示詞修改前後的對比。

#### 測試素材

```
主題：漲價焦慮
具體故事：上週有位學員，他之前的收費是每小時 1,800 元。跟我諮詢完之後，我建議他漲到 3,600 元，但他很擔心客戶會因此流失。我告訴他，這就是一種「創作者的焦慮」。事實上，漲價反而會讓大家更重視你的價值。雖然中間有 1,000 多元的價差，但其實每個人都是可以接受的。問題的核心在於：當價格提升、客戶付的錢變多之後，他們會更重視你的內容與影響力，這樣才能真正幫助到他們。
```

#### 測試類型

1. **故事型**：用這個素材寫一篇故事
2. **觀點型**：用這個素材寫一篇觀點
3. **對話型**：用這個素材寫一篇對話
4. **診斷型**：用這個素材寫一篇診斷

### 3.2 提示詞修改對比

#### 修改前（現有系統）

**System Prompt 特徵**：
- 總字數：~7,500 字
- 規則數量：30+ 條禁止規則
- Few-Shot：直接注入範文
- 開頭規則：強制使用「冠號斷言」
- CTA 規則：強制使用「你們覺得呢？」

#### 修改後（優化版本）

**System Prompt 特徵**：
- 總字數：~1,500 字
- 規則數量：5 條核心禁止規則
- Few-Shot：只注入風格特徵，不注入範文
- 開頭規則：提供多種選擇，不強制
- CTA 規則：根據內容類型動態選擇

### 3.3 優化後的提示詞範本

```typescript
const OPTIMIZED_SYSTEM_PROMPT = `

=== 創作者風格 DNA ===
${styleDNA}

3. 禁止說教結尾（「希望對你有幫助」）
4. 禁止過度語氣詞（每篇最多 2-3 個「超」「啊」「欸」）

直接輸出可發布的貼文，不要任何解釋。
`;
```

```
✅ 有轉折點
✅ 有情感變化
✅ 有故事結尾的啟發

【結構要求（英雄旅程架構）】：
```

```
- 保持故事的自然流動，不要刻意套結構
- 結尾可以是感悟、問句、或留白，不要公式化
```

## 七、結論

### 核心發現

1. **AI 感的根本原因不是缺少規則，而是規則太多**：7,500 字的提示詞讓 LLM 選擇性執行，反而產生公式化的內容。

2. **具體細節是降低 AI 感的關鍵**：當用戶提供具體的時間、人物、數字、對話時，AI 不需要編造，內容自然有「人味」。

3. **選題困難是創作者的真實痛點**：即使有 IP 地基和受眾資料，創作者也不知道如何轉化成具體的貼文主題。

### 建議實施順序

1. **立即**：提示詞精簡（效果最快、風險最低）
2. **短期**：輸入欄位優化（引導用戶提供具體細節）
3. **中期**：爆款選題庫（解決選題困難）
4. **長期**：持續 A/B 測試和優化

### 成功指標

- AI 感降低 40-60%
- 用戶生成滿意度提升 20%
- 需要修改次數減少 50%
- 選題困難問題解決
