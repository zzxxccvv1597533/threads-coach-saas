# 各貼文類型結構差異分析 + 更具體的選題靈感設計

## 一、您的觀察非常正確

> **不是每篇貼文都要 Hook + 故事 + CTA，不然還是會有模板感。**

分析目前系統中的 10 種貼文類型，它們確實有不同的結構：

| 類型 | 結構代碼 | 實際結構 | 是否需要 CTA？ |
|------|----------|----------|----------------|
| **故事型** | narrative | Hook → 情境 → 轉折 → 啟發 | 可有可無 |
| **知識型** | golden | 問題 → 3 個重點 → 行動建議 | 需要 |
| **整理型** | list | 標題 → 條列 → 總結 | 可有可無 |
| **觀點型** | argument | 觀點 → 論點 → 討論 | 需要 |
| **對話型** | qa | 問題 → 對話 → 金句 | 可有可無 |
| **引用型** | reflection | 引用 → 反應 → 延伸 | 可有可無 |
| **反差型** | twist | 預期 → 反轉 → 解釋 | 可有可無 |
| **閒聊型** | free | 情緒 → 碎片 → 隨意結尾 | **不需要** |
| **提問型** | question | 問題 → 背景 → 開放 | 本身就是問題 |
| **投票型** | poll | 選項 → 說明 → 投票 | 本身就是投票 |
| **診斷型** | diagnosis | 症狀 → 標籤 → 解析 | 可有可無 |

### 關鍵發現

1. **閒聊型**：不需要 CTA，結尾可以很隨意
2. **提問型/投票型**：本身就是互動形式，不需要額外 CTA
3. **故事型**：重點是轉折，不是 CTA
4. **觀點型**：需要討論，所以需要 CTA

---

## 二、目前的問題：統一的結構原則

目前的提示詞可能有這樣的問題：

```
// 目前的通用規則（可能造成模板感）
所有貼文都要：
- 開頭有 Hook
- 中間有故事或論點
- 結尾有 CTA（「你們覺得呢？」）
```

**這就是為什麼所有貼文看起來都很像。**

---

## 三、解決方案：每種類型有專屬的結構指引

### 3.1 修改後的結構指引

不再使用「通用規則」，而是根據貼文類型給出專屬指引：

#### 故事型（narrative）
```
結構：時間 → 衝突 → 轉折 → 啟發
重點：轉折要有情緒變化
結尾：可以是反思，不一定要問問題
範例結尾：「這件事讓我明白...」「後來我才懂...」
```

#### 知識型（golden）
```
結構：問題 → 3 個重點 → 行動建議
重點：每個重點要能獨立成金句
結尾：給具體的行動建議
範例結尾：「下次遇到這種情況，你可以...」
```

#### 閒聊型（free）
```
結構：情緒 → 碎片 → 隨意
重點：不需要結構，像在傳訊息
結尾：可以沒有結尾，或很隨意
範例結尾：「就這樣。」「好累。」「晚安。」
```

#### 提問型（question）
```
結構：問題 → 背景（選填）
重點：問題要好回答
結尾：本身就是問題，不需要額外 CTA
範例結尾：「你們覺得呢？」（這是問題本身，不是 CTA）
```

#### 觀點型（argument）
```
結構：觀點 → 2-3 個論點 → 討論
重點：觀點要有立場
結尾：需要引發討論
範例結尾：「你怎麼看？」「你同意嗎？」
```

### 3.2 技術實作

目前系統已經有 `aiPromptTemplate` 欄位，每種類型有專屬的提示詞模板。

**需要修改的是**：確保這些專屬模板被正確使用，而不是被「通用規則」覆蓋。

```typescript
// 修改前：通用規則 + 類型模板
const systemPrompt = `
${通用規則}  // 這裡有「所有貼文都要 CTA」
${類型專屬模板}  // 這裡說「閒聊型不需要 CTA」
`;
// 問題：通用規則和類型模板互相矛盾

// 修改後：只用類型專屬模板
const systemPrompt = `
${類型專屬模板}  // 只有這個
`;
// 效果：每種類型有自己的結構
```

## 四、更具體的選題靈感設計

### 4.1 您的觀察

> **選題靈感可以更具體，不只是給一個「主題」，而是可以給出更具體的「切角」或「開頭範例」。**

### 4.2 目前的設計（太抽象）

```
┌─────────────────────────────────────────────────────┐
│ 🔥 感情迷惘                                          │
│ 爆文率 68% · 互動率 12.3%                           │
│ 「很多人問我，他到底愛不愛我...」                    │
└─────────────────────────────────────────────────────┘
```

**問題**：
- 只給了一個「主題」（感情迷惘）
- 用戶還是不知道「具體要寫什麼」

### 4.3 更具體的設計

#### 方案 A：給出「切角」而非「主題」

```
┌─────────────────────────────────────────────────────┐
│ 💔 感情迷惘                                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│ 這個主題有 3 個熱門切角：                            │
│                                                     │
│ ┌─────────────────────────────────────────────┐    │
│ │ 切角 1：「他到底愛不愛我」的判斷方法          │    │
│ │ 類型：知識型 · 爆文率 72%                    │    │
│ │ 開頭範例：「3 個跡象判斷他是不是真的愛你」    │    │
│ └─────────────────────────────────────────────┘    │
│                                                     │
│ ┌─────────────────────────────────────────────┐    │
│ │ 切角 2：「放不下」的心理分析                  │    │
│ │ 類型：診斷型 · 爆文率 65%                    │    │
│ │ 開頭範例：「如果你有這 3 個症狀，你可能是...」│    │
│ └─────────────────────────────────────────────┘    │
│                                                     │
│ ┌─────────────────────────────────────────────┐    │
│ │ 切角 3：「我也曾經放不下」的故事              │    │
│ │ 類型：故事型 · 爆文率 58%                    │    │
│ │ 開頭範例：「三年前，我也是那個放不下的人...」 │    │
│ └─────────────────────────────────────────────┘    │
│                                                     │
│ [使用這個切角] [換一批]                              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**優點**：
- 給出具體的「切角」，不只是「主題」
- 每個切角有對應的「貼文類型」
- 每個切角有「開頭範例」，用戶知道要寫什麼

#### 方案 B：AI 生成「具體化的選題」

- 用戶可以直接選擇，不需要自己想

## 五、這樣做是否符合平台初衷？

### 平台初衷

> **讓命理老師或身心靈老師，不需要研究爆款選題、不需要學習寫作技巧，依然能取得好成績。**

### 分析

| 需求 | 方案 A（給切角） | 方案 B（AI 生成選題） |
|------|------------------|----------------------|
| 不需要研究爆款選題 | ✅ 系統給切角 | ✅ AI 生成選題 |
| 不需要學習寫作技巧 | ✅ 系統給開頭範例 | ✅ AI 給開頭 |
| 能取得好成績 | ⚠️ 切角來自爆款資料庫 | ✅ AI 結合爆款趨勢 |

**建議**：方案 B 更符合初衷，因為 AI 可以結合：
1. 用戶的領域（塔羅、命理、身心靈）
2. 爆款資料庫的趨勢
3. 用戶的 IP 地基（人設、風格）

生成「量身定制」的選題建議。

## 六、技術實作方案

### 6.1 新增 API：選題推薦

```typescript
// server/routers.ts
topicInspiration: protectedProcedure
  .input(z.object({
    count: z.number().default(3),
    refresh: z.boolean().default(false)
  }))
  .mutation(async ({ ctx, input }) => {
    // 1. 取得用戶的 IP 地基
    const ipBase = await getIpBase(ctx.user.id);
    
    // 2. 取得用戶領域的爆款資料
    const viralPosts = await getViralPostsByDomain(ipBase.domain);
    
    // 3. 用 AI 生成具體選題
    const topics = await generateTopicInspiration({
      domain: ipBase.domain,
      persona: ipBase.persona,
      viralPosts: viralPosts.slice(0, 10),
      count: input.count
    });
    
    return topics;
  })
```

### 6.2 AI 生成選題的提示詞

```typescript
const generateTopicInspiration = async (params) => {
  const prompt = `

${params.viralPosts.map(p => `- ${p.topic}: ${p.hook}`).join('\n')}

請生成 ${params.count} 個具體的選題建議，每個選題包含：
1. 選題標題（具體、有吸引力）

3. 切角說明（這個選題要從什麼角度切入）

注意：
- 選題要具體，不要太抽象

`;
```

## 七、總結

### 7.1 結構差異

每種貼文類型應該有專屬的結構指引，不是統一的「Hook + 故事 + CTA」。

| 類型 | 結構 | 結尾 |
|------|------|------|
| 故事型 | 時間 → 衝突 → 轉折 → 啟發 | 反思 |
| 知識型 | 問題 → 3 個重點 → 行動建議 | 行動建議 |
| 閒聊型 | 情緒 → 碎片 → 隨意 | 可以沒有 |
| 提問型 | 問題 → 背景 | 問題本身 |
| 觀點型 | 觀點 → 論點 → 討論 | 討論 |

### 7.2 選題靈感

從「給主題」升級為「給具體選題」：
- 選題標題（具體）
- 貼文類型（已選定）
- 切角說明（怎麼切入）
- 開頭範例（第一句話）

### 7.3 工作量估算

| 項目 | 時間 |
|------|------|
| 修改提示詞，移除通用規則 | 2-3 小時 |
| 新增選題推薦 API | 4-6 小時 |
| 前端選題靈感介面 | 6-8 小時 |
| **總計** | **12-17 小時** |
