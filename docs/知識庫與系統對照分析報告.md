# Threads 經營核心知識庫與系統對照分析報告

**報告日期**：2026-01-17  
**分析目的**：對照知識庫定義的內容類型與現有系統分類，找出差異並提出優化建議

---

## 一、知識庫定義的四種核心文章類型

根據 Threads 經營核心知識庫 PDF，內容創作應遵循以下四種核心類型：

| 類型 | 英文 | 核心目標 | 內容方向 | 適用情境 |
|------|------|---------|---------|---------|
| **故事型** | Story | 分享個人經歷，拉近與讀者的距離，建立信任感與共鳴 | 失敗經驗、心路歷程、生活啟發 | 想建立鮮明個人品牌、強調真實性的創作者 |
| **系列型** | Series | 鎖定一個主題進行深度連載，降低每日選題難度，培養讀者黏著度 | 主題教學、每日紀錄、案例拆解 | 知識體系完整、或想針對特定主題深入經營的創作者 |
| **知識型** | Knowledge | 分享專業知識或實戰經驗，為讀者提供具體價值，建立專業形象 | 工作技巧、產業觀察、經驗之談 | 各領域的專家、工作者或有實戰經驗的人 |
| **整理型** | Curation | 為讀者整理、篩選有用的資訊，節省時間，成為「資訊樞紐」 | 工具清單、資源列表、技巧總結 | 喜歡做筆記、整理資料、樂於分享資源的創作者 |

### 3.2 關鍵差異分析

**問題一：系列型（Series）完全缺失**

知識庫明確定義「系列型」為四大核心類型之一，但現有系統完全沒有對應。系列型的特點是：
- 主題連載（如「#泰國旅遊必知 10 件事」）
- 每日紀錄（如「#我的漲粉日記 Day X」）
- 案例拆解（將複雜案例分成數天分析）

**問題二：整理型（Curation）被歸入知識型**

爆款分類腳本將「整理型」內容（工具清單、資源列表、懶人包）歸類為「知識型」，但知識庫明確將兩者區分：
- 知識型：分享專業知識或實戰經驗
- 整理型：整理、篩選有用的資訊

**問題三：閒聊型佔比過高（76%）**

爆款分類結果顯示 76% 的文章被歸類為「閒聊型」，這可能是因為：
1. 分類規則過於寬鬆（預設規則將 200-250 字的文章都歸為閒聊型）
2. 缺少對「幽默型」的識別（幽默型平均讚數最高，但未被分類）
3. 缺少對「反差型」的識別

**問題四：提示詞系統與分類腳本不一致**

提示詞系統有 12 種類型，但分類腳本只有 6 種，導致：
- 幽默型（成功率最高 8.5%）無法被識別
- 反差型（成功率 5.5%）無法被識別
- 對話型、引用型無法被識別

## 四、爆款分類規則詳細檢視

現有分類腳本使用 9 條規則進行分類：

| 規則 | 判斷條件 | 分類結果 | 問題分析 |
|------|---------|---------|---------|
| 規則 1 | 開頭有問號且 < 300 字，或有投票詞彙 | interaction | ✅ 合理 |
| 規則 2 | 有列表格式且 > 350 字，或有教學詞彙且開頭有數字 | knowledge | ⚠️ 門檻過高（350 字） |
| 規則 3 | 包含診斷相關詞彙 | diagnostic | ✅ 合理 |
| 規則 4 | 以「我」開頭且有故事詞彙，或有情緒詞彙且 > 300 字 | story | ⚠️ 門檻過高（300 字） |
| 規則 5 | 有反差句式或觀點詞彙 | opinion | ✅ 合理 |
| 規則 6 | < 200 字且無列表、無教學詞彙 | casual | ⚠️ 過於寬鬆 |
| 規則 7 | 以「我」開頭且 > 250 字 | story | ✅ 合理 |
| 規則 8 | 有「你」相關句式且有問號 | interaction | ✅ 合理 |
| 規則 9 | 有 CTA 且有列表 | knowledge | ✅ 合理 |
| 預設 | > 400 字 → knowledge，> 250 字 → story，其他 → casual | - | ⚠️ 過於依賴字數 |

**主要問題**：
1. 缺少對「幽默型」的識別規則（如：自嘲、吐槽、笑死、誰懂）
2. 缺少對「反差型」的識別規則（如：不是...而是、真相是）
3. 缺少對「整理型」的識別規則（如：懶人包、N 個推薦、清單）
4. 缺少對「系列型」的識別規則（如：Day X、#系列標籤、Part X）
5. 預設規則過於依賴字數，導致大量文章被歸為「閒聊型」

## 五、優化建議

### 5.1 短期優化（分類規則調整）

**建議一：新增幽默型識別規則**

```typescript
// 規則 X：幽默型（自嘲、吐槽、反轉）
const hasHumorPattern = /笑死|誰懂|不要笑|我承認|說實話|太扯|傻眼|崩潰|受夠了|真的很好笑/.test(text);
const hasSelfMockPattern = /我真的|我就是|我承認我|我太|我好|我超/.test(opener);

if (hasHumorPattern && length < 200) {
  return "humor";
}
```

**建議二：新增整理型識別規則**

```typescript
// 規則 X：整理型（清單、懶人包、推薦）
const hasCurationPattern = /懶人包|推薦|清單|整理|收藏|必備|必看|必學|N個/.test(text);
const hasListWithTitle = /[1-9][\.、\)].*[1-9][\.、\)].*[1-9][\.、\)]/.test(text);

if ((hasCurationPattern && hasListPattern) || hasListWithTitle) {
  return "summary";
}
```

**建議三：新增系列型識別規則**

```typescript
// 規則 X：系列型（連載、每日紀錄）
const hasSeriesPattern = /Day\s*\d|第\d+天|#\w+\d|Part\s*\d|EP\s*\d|Vol\.\s*\d/.test(text);
const hasHashtagSeries = /#[^\s]+\d+/.test(text);

if (hasSeriesPattern || hasHashtagSeries) {
  return "series";
}
```

**建議四：調整預設規則**

將預設規則從「依賴字數」改為「依賴內容特徵」：

```typescript
// 預設規則優化
if (hasEmotionPattern) {
  return "story";  // 有情緒詞彙傾向故事型
} else if (startsWithI) {
  return "story";  // 以「我」開頭傾向故事型
} else if (hasYouPattern) {
  return "interaction";  // 有「你」傾向互動型
} else {
  return "casual";  // 真正的閒聊型
}
```

### 5.2 中期優化（類型體系統一）

**建議一：統一提示詞系統與分類腳本的類型定義**

將爆款分類腳本的 6 種類型擴充為 12 種，與提示詞系統一致：

| 新增類型 | 識別特徵 |
|---------|---------|
| humor（幽默型） | 自嘲、吐槽、笑死、誰懂 |
| contrast（反差型） | 不是...而是、真相是、其實 |
| dialogue（對話型） | 對話引用、「」開頭 |
| quote（引用型） | 引用名言、某人說過 |
| poll（投票型） | A/B 選、投票、你選哪個 |
| series（系列型） | Day X、#系列標籤、Part X |

**建議二：新增 content_type 欄位到資料庫**

目前已完成，但需要重新執行分類以使用新規則。

### 5.3 長期優化（AI 輔助分類）

**建議一：使用 LLM 進行二次分類校正**

對於規則分類結果為「閒聊型」的文章，使用 LLM 進行二次分類：

```typescript
const prompt = `
請判斷以下 Threads 貼文屬於哪種類型：
1. 故事型：分享個人經歷
2. 知識型：分享專業知識
3. 整理型：整理清單或懶人包
4. 觀點型：表達個人觀點
5. 幽默型：用幽默方式表達
6. 閒聊型：輕鬆日常分享

貼文內容：${postText}

請只回答類型名稱。
`;
```

**建議二：建立人工校正機制**

在管理後台新增「分類校正」功能，讓教練可以手動調整分類錯誤的文章，並將校正結果用於優化分類規則。

## 六、總結

### 現有系統的優點
1. 提示詞系統定義了 12 種細緻的貼文類型，涵蓋面廣
2. 每種類型都有詳細的字數範圍、推薦開頭、結構模板
3. 基於 17,850 篇爆款資料庫分析，有數據支撐

### 需要改進的地方
1. **系列型缺失**：知識庫四大核心類型之一，但系統完全沒有對應
2. **分類規則過於依賴字數**：導致 76% 文章被歸為閒聊型
3. **高效類型未被識別**：幽默型（平均讚數最高）、反差型未被分類
4. **提示詞系統與分類腳本不一致**：12 種 vs 6 種

### 建議優先級
1. **高優先**：新增幽默型、整理型識別規則
2. **中優先**：新增系列型識別規則
3. **低優先**：統一類型體系、建立人工校正機制
