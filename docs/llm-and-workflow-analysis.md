# LLM 等級、品質檢查、與 Prompt 流程分析報告

## 一、LLM 等級與生成品質的關係

### 目前使用的模型

| 項目 | 值 |
|------|-----|
| **模型名稱** | `gemini-2.5-flash-preview-09-2025` |
| **提供者** | Google（透過 Manus Forge API 代理） |
| **特性** | 高效能、低成本、支援長上下文 |

### 更高級別 LLM 的潛在效益

| 模型等級 | 代表模型 | 預期效益 | 成本增加 |
|----------|----------|----------|----------|
| **當前（Flash）** | Gemini 2.5 Flash | 基準線 | 1x |
| **中階（Pro）** | Gemini 2.5 Pro / GPT-4o | 指令遵循度 +15%、創意度 +20% | 3-5x |
| **高階（Reasoning）** | o1 / Claude 3.5 Opus | 複雜推理 +30%、風格一致性 +25% | 10-20x |

### 建議策略

**分級使用**：
- **開頭生成（5 個候選）**：使用 Flash（成本敏感）
- **正文生成**：使用 Pro（品質優先）
- **品質檢查/重寫**：使用 Flash（規則驅動）

這樣可以在控制成本的同時提升關鍵環節的品質。

## 二、品質檢查失敗原因分析

### 測試結果回顧

| 測試內容 | 通過 | 分數 | 失敗原因 |
|----------|------|------|----------|
| 優質內容 | ✅ | 89 | - |
| 含禁止句式 | ❌ | 87 | 第一層偵測到「首先」「讓我們」 |
| AI 痕跡明顯 | ❌ | 61 | 第一層偵測到「毋庸置疑」等 |

### 失敗原因解析

**第一層檢查（Prompt 注入禁止句式）** 是最嚴格的檢查層，它會偵測以下類型的禁止句式：

1. **AI 常用開頭詞**：「首先」「其次」「最後」「讓我們」「一起來」
2. **AI 常用連接詞**：「毋庸置疑」「不得不說」「值得一提」
3. **AI 常用結尾詞**：「希望對你有幫助」「總而言之」

### 這是「預期行為」還是「問題」？

**這是預期行為**。品質檢查的目的就是偵測這些 AI 痕跡明顯的內容，並提供修復建議。

測試中的「含禁止句式」和「AI 痕跡明顯」內容本來就應該被標記為失敗，因為它們確實包含了需要修正的問題。

### 系統的自動修復機制

當內容被標記為失敗時，系統會：

1. **提供具體建議**：告訴用戶哪些詞需要移除或替換
2. **自動修復**：嘗試自動替換禁止句式
3. **重新檢查**：修復後重新計算品質分數

## 三、引導模式與精簡模式的運作流程

### 目前系統的兩種主要模式

根據程式碼分析，系統目前有以下模式：

#### 1. 沒靈感模式（腦力激盪）

**API**: `brainstorm`

**流程**：
```
用戶選擇內容支柱/主題
    ↓
系統讀取 IP 地基、受眾、產品資料
    ↓
LLM 生成 5 個選題建議
    ↓
每個建議包含：標題、流量密碼、Hook 類型、目標情緒、CTA
```

#### 2. 有素材模式（草稿生成）

**API**: `generateDraft`

**流程**：
```
用戶輸入素材 + 選擇內容類型
    ↓
系統收集上下文：
  - IP 地基（職業、語氣、人設三支柱、英雄旅程）
  - 受眾資料（痛點、渴望）
  - 用戶風格（語氣、口頭禪、範文）
  - 爆文因子（市場數據）
  - Few-Shot 範例（分層範例庫）
  - 數據驅動開頭模式
    ↓
根據 editMode 調整策略：
  - light（輕度優化）：幾乎不改變內容
  - preserve（風格保留）：保留敘事結構
  - rewrite（爆款改寫）：完整套用爆款公式
    ↓
LLM 生成內容
    ↓
後處理：
  - 清理 AI 標記
  - 應用去 AI 化過濾器
  - 字數檢查
    ↓
儲存草稿
```

### 開頭生成流程（新增優化）

**API**: `generateMultipleOpeners`

**流程**：
```
收集上下文（IP 地基、用戶風格、開頭 DNA）
    ↓
判斷用戶成長階段（專家/成熟/成長/新手）
    ↓
計算自適應品質門檻
    ↓
選擇範例比例（系統範例 vs 用戶範例）
    ↓
生成 5 個開頭候選
    ↓
三層品質檢查：
  1. Prompt 注入檢查
  2. AI Detector
  3. 品質分數計算
    ↓
自動修復（如果需要）
    ↓
排序並返回最佳候選
```

## 四、優化前後 Prompt 運作交叉比對

### 4.1 品質門檻機制

| 項目 | 優化前 | 優化後 |
|------|--------|--------|
| **門檻類型** | 固定值（50 互動） | 自適應（相對值 + 絕對下限） |
| **用戶階段判斷** | 無 | 專家/成熟/成長/新手 四級 |
| **範例選擇** | 固定比例 | 動態比例（專家 90% 用戶/新手 70% 系統） |
| **低品質處理** | 全部使用 | 僅用於風格分析，不用於生成 |

### 4.2 品質檢查機制

| 項目 | 優化前 | 優化後 |
|------|--------|--------|
| **檢查層數** | 1 層（AI Detector） | 3 層（Prompt 注入 + AI Detector + 品質分數） |
| **自動修復** | 無 | 有（最多 3 次重試） |
| **品質分數** | 無 | 有（0-100 分，分 4 級） |
| **建議系統** | 無 | 有（具體修改建議） |

### 4.3 開頭生成機制

| 項目 | 優化前 | 優化後 |
|------|--------|--------|
| **開頭 DNA** | 無 | 有（提取用戶成功模式） |
| **最近使用追蹤** | 無 | 有（避免重複） |
| **動態權重** | 無 | 有（根據用戶資料量調整） |

### 4.4 Prompt 長度控制

| 項目 | 優化前 | 優化後 |
|------|--------|--------|
| **長度限制** | 無 | 8,000 字符上限 |
| **智能截斷** | 無 | 有（優先保留核心指令） |
| **長度監控** | 無 | 有（記錄到指標收集器） |

### 4.5 快取機制

| 項目 | 優化前 | 優化後 |
|------|--------|--------|
| **用戶風格快取** | 無 | 5 分鐘 TTL |
| **爆文模式快取** | 無 | 10 分鐘 TTL |
| **禁止句式快取** | 無 | 30 分鐘 TTL |

## 六、總結與建議

### 目前系統狀態

1. **LLM**：使用 Gemini 2.5 Flash，適合大量生成任務
2. **品質檢查**：三層檢查機制正常運作，失敗是預期行為
3. **流程**：引導模式和精簡模式都已整合新的優化功能

### 建議的下一步

1. **LLM 分級使用**：考慮在正文生成時使用更高級別的模型
2. **A/B 測試**：對比優化前後的品質分數和用戶滿意度
3. **監控儀表板**：建立前端介面展示指標數據

### 需要修復的問題

**無需修復**。品質檢查失敗是預期行為，系統正確地偵測到了 AI 痕跡明顯的內容。

如果您希望調整檢查的嚴格程度，可以修改 `shared/rules/quality-rules.ts` 中的規則。
