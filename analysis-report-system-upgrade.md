**報告日期**：2026 年 2 月 1 日

## 一、執行摘要

本報告針對兩份系統升級方案文件進行全面分析，評估將現有「孤島式工具」整合為「自我改善閉環系統」的可行性。經過詳細評估，我們的結論是：**這些修改整體上是適合的，但需要分階段實施，並在特定領域做出調整以符合幕創行銷的產品定位與用戶需求**。

核心發現如下：

| 評估維度 | 評估結果 | 說明 |
|---------|---------|------|
| **技術可行性** | ✅ 高 | 現有模組（embedding、prompt builder、diagnostics）已具備整合基礎 |
| **產品價值** | ✅ 高 | 能顯著提升內容品質、降低用戶修稿時間、實現個人化 |
| **實施風險** | ⚠️ 中等 | 系統複雜度提升、成本增加、需要完善的治理機制 |
| **與現有系統相容性** | ✅ 高 | 方案設計已考慮現有架構，可漸進式整合 |
| **用戶體驗影響** | ⚠️ 需注意 | 過度指導可能壓抑創意，需設計適當的「退出機制」 |

## 二、方案概述與核心理念

### 2.1 方案的第一性原理

升級方案基於三個核心原則：

1. **保護創作者的聲音（自主性）**：確保 AI 是「助手」而非「代筆者」
2. **提供有學習價值的回饋（有效回饋）**：每次互動都能幫助創作者成長
3. **用真實數據檢驗並自我改善（閉環）**：系統能根據實際表現持續優化

### 2.2 方案的核心架構

方案提出將現有分散的模組整合為一條「生成 → 偵測 → 自動修正 → 記錄結果 → 以結果調整下次生成」的閉環流水線：

```
┌─────────────────────────────────────────────────────────────────┐
│                        閉環系統架構                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │ 向量 DB  │───▶│ Prompt   │───▶│ 內容生成 │───▶│ 健康檢測 │  │
│  │ 語意檢索 │    │ Builder  │    │ + 潤飾   │    │ + AI痕跡 │  │
│  └──────────┘    └──────────┘    └──────────┘    └────┬─────┘  │
│       ▲                                               │        │
│       │                                               ▼        │
│  ┌────┴─────┐                                   ┌──────────┐   │
│  │ 策略權重 │◀──────────────────────────────────│ 用戶採納 │   │
│  │ 調整     │                                   │ + 成效   │   │
│  └──────────┘                                   └──────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、各項修改的詳細評估

### 3.1 短期快贏項目（P0/P1，1-3 週）

#### 3.1.1 向量資料庫（Vector DB）上線

| 評估項目 | 評估內容 |
|---------|---------|
| **目的** | 讓 Embedding 能在生成時做即時語意檢索、同質性檢查與多樣性重排（MMR） |
| **技術方案** | pgvector / Milvus / Pinecone，實作 `semanticSearch` 與 `mmr_search` API |
| **適合度** | ⭐⭐⭐⭐⭐ 非常適合 |

**分析**：

向量資料庫的導入是整個升級方案的基石。目前系統已有 `viralExampleEmbeddings` 與 `userPostEmbeddings` 的生成邏輯，但僅用於本地/批次比對。將其上架為即時查詢服務，能帶來三個直接好處：

1. **選題推薦更精準**：根據用戶輸入的主題，語意搜尋最相關的爆款範例
2. **開頭選擇更多樣**：使用 MMR（Maximal Marginal Relevance）確保推薦的開頭風格多樣化
3. **同質化檢查自動化**：生成後立即比對庫中近鄰，若相似度過高自動觸發重寫

**建議閾值**：
- 過度相似閾值：cosine similarity > 0.88 → 觸發重寫
- 語意保真度閾值：0.80（潤飾檢查）

**風險與對策**：
- **延遲風險**：向量檢索會增加 API 回應時間，建議設計快取機制（熱門主題預先檢索）
- **成本風險**：選擇 pgvector（開源）作為初期方案，降低成本

---

#### 3.1.2 recommendedHooks 與 stylePolish 整合到 Prompt Builder

| 評估項目 | 評估內容 |
|---------|---------|
| **目的** | 生成時利用有證據的高效鉤子，並確保潤飾不改寫原意 |
| **技術方案** | 在 `buildDataDrivenSystemPrompt()` 中注入 top-3 recommendedHooks，實作語意保真檢查 |
| **適合度** | ⭐⭐⭐⭐⭐ 非常適合 |

**分析**：

目前 `recommendedHooks` 僅存在於內部日誌，未被實際使用於生成流程。將其整合到 Prompt Builder 是「低成本高收益」的改動：

1. **Few-shot 不再隨機**：從「隨機抽取」升級為「語意搜尋 + 品質分層」
2. **語意保真檢查**：確保潤飾後的內容與原稿語意相似度 ≥ 0.80

**重要提醒**：在注入 few-shot 時，必須加入明確指令：「請參考下面 3 個開頭風格，感受其節奏與關鍵元素，但不要直接複製句子。」這能有效降低抄襲感，符合幕創行銷「保護創作者聲音」的核心理念。

**與現有系統的相容性**：
- `data-driven-prompt-builder.ts` 已存在 `getSmartFewShotExamples` 函數，可直接擴展
- `buildStylePolishSystemPrompt` 已存在，只需增加語意保真檢查邏輯

---

#### 3.1.3 quickDetect / content-health-check 嵌入生成流（Online Guardrail）

| 評估項目 | 評估內容 |
|---------|---------|
| **目的** | 讓診斷不再是手動後處理，而成為生成「即時反饋」的一部分 |
| **技術方案** | 初稿生成後同步呼叫健康檢測，若未達標自動要求 LLM 修正 |
| **適合度** | ⭐⭐⭐⭐ 適合，但需設計「退出機制」 |

**分析**：

這是提升「初稿可直接接受比例」的關鍵改動。目前的流程是：生成 → 用戶手動檢查 → 手動修改。改為：生成 → 自動檢測 → 自動修正 → 用戶確認。

**建議的檢測閾值**：
- AI 痕跡數量 > 3 → 觸發自動修正
- health_score < 70 → 觸發自動修正

**風險與對策**：

| 風險 | 對策 |
|-----|------|
| 誤判造成不必要的修正 | 設計「跳過自動修正」按鈕，讓用戶有最終決定權 |
| 自動修正迴圈過長 | 限制最多 2 次自動修正，超過則交由用戶決定 |
| 延遲增加 | 使用串流（streaming）方式，讓用戶在等待時能看到進度 |

**特別注意**：根據幕創行銷的知識庫，「AI 的角色應該是助手，而非替代者」。因此，自動修正功能必須設計為「建議」而非「強制」，讓創作者保有最終決定權。

### 3.2 中期系統化整合項目（P1，1-3 個月）

#### 3.2.1 userInteractionEvents 與個人化 Signal

| 評估項目 | 評估內容 |
|---------|---------|
| **目的** | 讓 AI「有記性」，根據使用者過去採用的建議與編輯習慣自適應建議 |
| **技術方案** | 建立 `userInteractionEvents` 表，計算 `suggestion_adoption_rate` 與 `user_edit_profile` |
| **適合度** | ⭐⭐⭐⭐⭐ 非常適合，是「教練系統」的核心 |

**分析**：

這是從「工具」升級到「教練系統」的關鍵。目前系統無法記住用戶的偏好，每次生成都是「從零開始」。建立個人化 Signal 後，系統能：

1. **記住用戶偏好**：「此用戶常刪除過於銷售的 CTA；偏好閒聊語氣」
2. **調整建議策略**：減少用戶不喜歡的建議類型，增加用戶偏好的類型
3. **實現「越用越懂你」**：隨著互動增加，建議越來越精準

**建議的 Schema**：

```sql
CREATE TABLE userInteractionEvents (
  id SERIAL PRIMARY KEY,
  userId VARCHAR(255) NOT NULL,
  sessionId VARCHAR(255) NOT NULL,
  suggestionId VARCHAR(255),
  suggestionType VARCHAR(50),
  taken BOOLEAN DEFAULT FALSE,
  editsCount INT DEFAULT 0,
  published BOOLEAN DEFAULT FALSE,
  postId VARCHAR(255),
  timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**與現有系統的整合**：
- 可在 `writingSession.complete` 時記錄用戶是否採用建議
- 可在 `generateDraft` 時注入 `userPreferenceContext`

#### 3.2.2 Post Metrics 與 AB 實驗結果作為策略權重

| 評估項目 | 評估內容 |
|---------|---------|
| **目的** | 用真實表現信號調整生成器的優先策略 |
| **技術方案** | ETL 關聯 postMetrics 與 generationConfig，用回歸模型評估相關性 |
| **適合度** | ⭐⭐⭐⭐ 適合，但需注意「資料品質」問題 |

**分析**：

這是實現「閉環優化」的關鍵。目前系統的策略是「寫死的規則」，無法根據實際表現調整。導入策略權重後，系統能：

1. **識別高效配置**：哪種 hook + 哪種內容類型 → 高互動
2. **動態調整權重**：A 型 hook 歷史上產生 2x comments → 提升 A 型 hook 權重
3. **科學化決策**：用數據而非直覺決定策略

**風險與對策**：

| 風險 | 對策 |
|-----|------|
| 資料品質問題（垃圾進垃圾出） | 設計資料品質檢查，過濾異常值 |
| 過度擬合（noise 放大成策略） | 使用半自動（人 + 機器）審核，設定最小樣本量 |
| 同質化加劇（大家都用高效配置） | 設計多樣化策略，強制輪替機制 |

---

#### 3.2.3 userStage + Humanizer 動態策略

| 評估項目 | 評估內容 |
|---------|---------|
| **目的** | 根據用戶成熟度動態調整 Humanizer 的嚴格度與教練強度 |
| **技術方案** | 用 userGrowthMetrics 判定 userStage（new / growing / mature），動態調整閾值 |
| **適合度** | ⭐⭐⭐⭐⭐ 非常適合，符合「因材施教」理念 |

**分析**：

這是解決「一刀切問題」的關鍵。目前 Humanizer 的 24 種規則對所有用戶一視同仁，但：

- **新手**：需要嚴格保護，避免產出明顯的 AI 痕跡
- **成熟用戶**：需要創作空間，過度干預會壓抑創意

**建議的 userStage 定義**：

| 階段 | 判定條件 | Humanizer 策略 |
|-----|---------|---------------|
| **new** | 創作次數 < 10 | 嚴格過濾（AI 痕跡容忍閾低） |
| **growing** | 創作次數 10-50，自主修改率 > 30% | 中等過濾 |
| **mature** | 創作次數 > 50，自主修改率 > 50% | 寬鬆但保留核心禁止項 |

## 四、buildPromptByMode 三種模式評估

方案提出的三種模式設計非常符合幕創行銷的產品定位：

| 模式 | 說明 | System Prompt 內容 | 適用場景 |
|-----|------|-------------------|---------|
| **pure_story** | 純個人故事 | 僅放寫作規則與 userStyle，不注入 IP / viral examples | 用戶想要完全原創、不受爆款模板影響 |
| **light_connect** | 輕度連結 | 注入 IP 地基（人設三支柱）但不注入 few-shot viral examples | 用戶想要保持人設一致性，但不想被爆款套路 |
| **full_inject** | 完整導入 | 注入所有（IP + viral data + success factors） | 用戶想要最大化互動率，願意參考爆款模板 |

**與現有系統的對應**：

這三種模式與現有的「創作意圖」（純粹分享 / 順便帶點專業 / 專業引導）可以形成對應：

| 創作意圖 | 建議的 buildPromptByMode |
|---------|-------------------------|
| 純粹分享 | pure_story |
| 順便帶點專業 | light_connect |
| 專業引導 | full_inject |

**驗收要點**：
- `pure_story` 模式產出的 system prompt 不可有 IP/爆款關鍵字（可用自動單元測試驗證）

## 七、結論與建議

### 7.1 整體評估結論

**這些修改整體上是適合的**，原因如下：

1. **技術可行性高**：現有模組已具備整合基礎，方案設計合理
2. **產品價值明確**：能顯著提升內容品質、降低用戶修稿時間、實現個人化
3. **與產品定位一致**：符合「AI 是助手，不是代筆者」的核心理念
4. **風險可控**：主要風險都有明確的對策

### 7.2 關鍵建議

1. **分階段實施**：先完成短期快贏項目，驗證效果後再推進中長期項目
2. **設計退出機制**：所有自動化功能都應該有「跳過」或「忽略」選項
3. **監控關鍵指標**：建立 dashboard 追蹤 AI 痕跡數量、採納率、修改次數等
4. **漸進式放寬**：根據 userStage 動態調整嚴格度，避免一刀切
5. **多樣化優先**：使用 MMR 和輪替機制，避免同質化加劇

### 7.3 需要特別注意的事項

1. **純個人故事模式（pure_story）必須真正「純淨」**：不能有任何 IP/爆款關鍵字洩漏
2. **語意保真檢查的閾值需要調校**：建議從 0.80 開始，根據用戶反饋調整
3. **自動修正迴圈需要限制**：最多 2 次，避免無限迴圈
4. **用戶偏好需要時間累積**：初期可能效果不明顯，需要耐心等待數據累積
