## 一、優化總覽與願景

### 1.1 優化目標

本次優化的核心目標是解決「**AI 感比較重**」的問題，同時提升系統的可擴展性和數據驅動能力。具體目標包括：

| 目標 | 說明 |
|-----|------|
| **降低 AI 感** | 讓生成的內容更自然、更像學員自己寫的 |
| **提升多樣性** | 避免開頭句式重複、內容結構雷同 |
| **保留爆款能量** | 在追求多樣性的同時，不犧牲內容的吸引力 |
| **數據驅動優化** | 建立回饋機制，讓系統能持續學習和改進 |
| **提升學員成績** | 幫助學員獲得更高的觸及、互動和轉換 |

## 二、第一階段：基礎建設（1-2 週）

### 2.1 優化項目清單

| 項目 | 說明 | 工時 |
|-----|------|------|
| **Data Storage** | 新增必要的 DB tables | 1-2 天 |
| **Prompt Service** | 集中管理 prompt 和 template | 2-3 天 |
| **Avoid-list** | 建立禁止句式清單 | 1 天 |
| **AI Detector（規則版）** | 檢測 AI 痕跡並標記 | 2-3 天 |
| **Observability（基礎）** | 建立 logging 和埋點 | 2 天 |

### 2.2 各項目的具體好處

#### 2.2.1 Data Storage（資料儲存）

**新增的表格**：

| 表格 | 用途 |
|-----|------|
| `opener_templates` | 儲存開頭模板和權重 |
| `prompt_avoid_list` | 儲存禁止句式清單 |
| `openers_candidates` | 儲存所有生成的候選 |
| `template_stats` | 儲存模板統計數據 |
| `ai_detector_logs` | 儲存 AI 檢測記錄 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **數據可追溯** | 所有生成記錄都有保存，可以回溯分析 |
| **支援 A/B 測試** | 可以比較不同模板的效果 |
| **持續學習基礎** | 為後續的機器學習提供訓練數據 |
| **問題診斷** | 當出現問題時可以快速定位原因 |

#### 2.2.2 Prompt Service（Prompt 管理服務）

**功能**：

| 功能 | 說明 |
|-----|------|
| `getActiveTemplates()` | 取得所有啟用的模板 |
| `getTemplateById(id)` | 取得特定模板 |
| `getAvoidList()` | 取得禁止句式清單 |
| `assemblePrompt(templateId, context)` | 組裝完整的 prompt |
| `updateTemplateWeight(id, weight)` | 更新模板權重 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **集中管理** | 所有 prompt 在一個地方管理，不散落在程式碼各處 |
| **版本控制** | 可以追蹤 prompt 的修改歷史 |
| **動態更新** | 無需重新部署即可更新 prompt |
| **A/B 測試** | 可以同時測試多個版本的 prompt |
| **權重調整** | 可以動態調整模板的使用頻率 |

#### 2.2.3 Avoid-list（禁止句式清單）

**預設禁止的句式**：

| 類型 | 範例 |
|-----|------|
| **開頭句式** | 「你是不是也這樣？」「你有沒有發現...」 |
| **過渡句式** | 「其實...」「說真的...」「不得不說...」 |
| **結尾句式** | 「你覺得呢？」「留言告訴我」 |
| **AI 特徵詞** | 「作為一個...」「首先...其次...最後...」 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **立即見效** | 實作簡單，效果明顯 |
| **可動態更新** | 發現新的 AI 痕跡可以即時加入 |
| **風險低** | 不會影響現有功能 |
| **用戶可參與** | 學員反饋的 AI 痕跡可以加入清單 |

#### 2.2.4 AI Detector（AI 痕跡檢測）

**檢測規則**：

| 規則 | 說明 |
|-----|------|
| **Avoid-list 匹配** | 檢查是否包含禁止句式 |
| **重複模式檢測** | 檢查是否有重複的句式結構 |
| **典型 AI 短語** | 檢查是否有常見的 AI 用語 |
| **句式密度** | 檢查特定句式的出現頻率 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **最後一道防線** | 在輸出前攔截 AI 痕跡 |
| **可量化追蹤** | 可以統計 AI 痕跡的出現頻率 |
| **用戶可見改善** | 學員能明顯感受到內容更自然 |
| **先標記不刪除** | 避免誤殺好內容，同時收集數據 |

#### 2.2.5 Observability（可觀測性）

**記錄的事件**：

| 事件 | 說明 |
|-----|------|
| `generate_request` | 用戶發起生成請求 |
| `candidate_written` | 候選寫入 DB |
| `ai_detected` | AI 痕跡被檢測到 |
| `user_selection` | 用戶選擇了某個候選 |
| `user_rewrite` | 用戶手動修改了內容 |
| `publish_event` | 內容被發布 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **數據驅動決策** | 基於真實數據做優化決策 |
| **效果可量化** | 可以追蹤每次改動的效果 |
| **問題快速定位** | 出問題時可以快速找到原因 |
| **持續優化基礎** | 為後續的機器學習提供數據 |

## 三、第二階段：智能生成（2-6 週）

### 3.1 優化項目清單

| 項目 | 說明 | 工時 |
|-----|------|------|
| **Opener Generator** | 生成多個開頭候選 | 3-5 天 |
| **簡單 Selector** | 規則式候選排序 | 2-3 天 |
| **ε-greedy Bandit** | 探索與利用平衡 | 2-3 天 |
| **前端候選選擇 UI** | 讓用戶選擇開頭 | 3-5 天 |
| **Resilience** | 外部服務容錯 | 2 天 |

### 3.2 各項目的具體好處

#### 3.2.1 Opener Generator（多候選生成）

**工作流程**：

```
用戶請求 → 選擇 3-5 個模板 → 每個模板生成 1-2 個候選 → 返回 6 個候選 → 用戶選擇
```

**好處**：

| 好處 | 說明 |
|-----|------|
| **保證多樣性** | 每次生成多個不同風格的候選 |
| **用戶有選擇權** | 學員可以選擇最符合自己風格的開頭 |
| **收集偏好數據** | 記錄用戶的選擇，學習用戶偏好 |
| **降低 AI 感** | 多樣化的候選減少重複感 |

#### 3.2.2 簡單 Selector（候選排序）

**排序因素**：

| 因素 | 權重 | 說明 |
|-----|------|------|
| **預測效果** | 40% | 基於歷史數據預測的爆款機率 |
| **多樣性** | 30% | 與最近生成內容的差異度 |
| **用戶風格匹配** | 20% | 與用戶 IP 地基的匹配度 |
| **AI 痕跡分數** | 10% | AI Detector 的檢測分數（越低越好） |

**好處**：

| 好處 | 說明 |
|-----|------|
| **智能排序** | 把最好的候選排在前面 |
| **平衡多目標** | 同時考慮效果、多樣性、風格 |
| **可解釋** | 可以告訴用戶為什麼推薦這個候選 |
| **可調整** | 可以根據效果調整各因素的權重 |

#### 3.2.3 ε-greedy Bandit（探索與利用平衡）

**策略**：

| 比例 | 行為 | 說明 |
|-----|------|------|
| **90%** | 利用（Exploitation） | 使用效果最好的模板 |
| **10%** | 探索（Exploration） | 嘗試新模板或低頻模板 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **自動優化** | 系統會自動偏向效果好的模板 |
| **持續創新** | 10% 的探索確保不會陷入局部最優 |
| **數據驅動** | 基於真實效果數據做決策 |
| **動態調整** | 可以根據情況調整探索比例 |

**UI 設計**：

## 四、第三階段：深度優化（2-4 月）

### 4.1 優化項目清單

| 項目 | 說明 | 工時 |
|-----|------|------|
| **AI Detector（Embedding）** | 基於向量的相似度檢測 | 5-7 天 |
| **學習式 Selector** | 機器學習排序模型 | 7-10 天 |
| **Training Pipeline** | 自動訓練流程 | 5-7 天 |
| **多樣性正則** | 全局多樣性控制 | 3-5 天 |
| **用戶風格 Profile 深化** | 深度風格學習 | 7-10 天 |

### 4.2 各項目的具體好處

#### 4.2.1 AI Detector（Embedding 版）

**檢測方式**：

| 方式 | 說明 |
|-----|------|
| **與 AI 句式庫的相似度** | 計算與已知 AI 句式的向量相似度 |
| **與用戶歷史的相似度** | 計算與用戶過去內容的相似度 |
| **與最近生成的相似度** | 避免短期內重複相似內容 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **更精準** | 向量相似度比規則匹配更準確 |
| **可發現新模式** | 能檢測出規則未覆蓋的 AI 痕跡 |
| **個人化** | 考慮用戶自己的風格 |
| **持續學習** | 可以不斷更新 AI 句式庫 |

#### 4.2.2 學習式 Selector

**模型特徵**：

| 特徵類型 | 特徵 |
|-----|------|
| **內容特徵** | 長度、情感分數、關鍵詞密度 |
| **模板特徵** | 模板歷史效果、使用頻率 |
| **用戶特徵** | 用戶風格 profile、歷史偏好 |
| **上下文特徵** | 發文時間、主題類別 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **更準確的預測** | 機器學習模型比規則更準確 |
| **自動發現模式** | 能發現人工難以發現的規律 |
| **持續改進** | 隨著數據增加，模型會越來越準 |
| **個人化推薦** | 可以針對每個用戶做個人化排序 |

#### 4.2.3 用戶風格 Profile 深化

**Profile 內容**：

| 維度 | 說明 |
|-----|------|
| **語言風格** | 用詞習慣、句式偏好、語氣特徵 |
| **內容偏好** | 喜歡的主題、常用的框架 |
| **開頭偏好** | 偏好的開頭類型、選擇歷史 |
| **效果模式** | 哪類內容效果最好 |
| **情緒詞彙** | 常用的情緒表達詞 |

**好處**：

| 好處 | 說明 |
|-----|------|
| **高度個人化** | 生成的內容更像用戶自己寫的 |
| **風格一致性** | 保持用戶 IP 的一致性 |
| **效果優化** | 基於用戶歷史效果優化生成 |
| **差異化** | 每個用戶的內容都有獨特風格 |
