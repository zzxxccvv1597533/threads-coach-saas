## 三、建議一：新增「診斷型」貼文類型

### 3.1 建議內容

在 `shared/content-types-v2.ts` 中新增 `diagnosis` 類型定義，讓用戶可以在前端選擇「診斷型」貼文。

### 3.2 優點分析

| 優點 | 說明 |
|------|------|
| **高互動潛力** | 診斷型貼文是 Threads 上互動率最高的類型之一，「你是哪一型？」這類問題天然引發留言 |
| **後端已就緒** | routers.ts 第 1399 行已有完整的診斷型 Prompt 邏輯，只差前端定義 |
| **差異化競爭** | 目前多數 AI 寫作工具沒有專門的診斷型模板，這是產品亮點 |
| **適合命理領域** | 診斷型非常適合命理師、塔羅師等用戶，與目標客群高度匹配 |

### 3.3 缺點分析

| 缺點 | 說明 |
|------|------|
| **開發成本低** | 只需新增約 50 行前端定義，無需後端修改 |
| **測試需求** | 需要測試 AI 生成的診斷型內容品質是否符合預期 |

### 3.4 建議結論

> **強烈建議執行** ⭐⭐⭐⭐⭐

理由：投入產出比極高，後端已完成，只需補前端定義即可啟用。這是「低成本高收益」的典型案例。

## 四、建議二：戰報閉環資料欄位擴充

### 4.1 建議內容

在 `postMetrics` 表中新增三個欄位：
- `hookType`：開頭類型（問題型、數字型、故事型等）
- `contentType`：內容類型（故事型、知識型、診斷型等）
- `ctaType`：結尾類型（提問型、行動型、分享型等）

### 4.2 優點分析

| 優點 | 說明 |
|------|------|
| **數據驅動優化** | 可以分析「哪種 Hook + 哪種內容 + 哪種 CTA」的組合最有效 |
| **AI 學習基礎** | 有了這些標籤，未來可以訓練 AI 自動推薦最佳組合 |
| **個人化建議** | 可以告訴用戶「你的故事型貼文平均觸及比知識型高 30%」 |
| **閉環完整性** | 這是真正實現「戰報 → 學習 → 優化」閉環的關鍵拼圖 |

### 4.3 缺點分析

| 缺點 | 說明 | 影響程度 |
|------|------|----------|
| **資料回填問題** | 現有貼文沒有這些欄位，需要決定如何處理歷史資料 | 中 |
| **標籤來源問題** | 這些標籤由誰填寫？用戶手動？AI 自動分析？ | 高 |
| **開發成本** | 需要修改 schema、API、前端三層，工作量較大 | 中 |
| **用戶負擔** | 如果要用戶手動選擇，會增加填寫戰報的摩擦 | 高 |

### 4.4 深度分析：標籤來源問題

這是最關鍵的問題。有三種可能的實作方式：

| 方式 | 優點 | 缺點 |
|------|------|------|
| **用戶手動選擇** | 資料準確 | 增加用戶負擔，可能導致填寫率下降 |
| **AI 自動分析** | 用戶無感 | 需要額外 API 成本，準確度不確定 |
| **從草稿繼承** | 無額外成本 | 只適用於從系統發布的貼文，外部貼文無法追蹤 |

### 4.5 建議結論

> **建議延後執行** ⭐⭐⭐

理由：這個功能的價值很高，但實作複雜度也高。建議先觀察以下數據再決定：

1. **目前有多少用戶在使用戰報功能？** 如果使用率低，先優化使用率比新增欄位更重要
2. **用戶是否有「想知道哪種類型表現好」的需求？** 可以先做用戶訪談
3. **是否有足夠的歷史數據？** 至少需要 100+ 筆有完整 metrics 的貼文才有分析價值

**替代方案**：可以先在現有的 `selfReflection` 欄位中引導用戶填寫類型資訊，觀察用戶反應後再決定是否正式新增欄位。

## 五、建議三：放寬排版規則

### 5.1 建議內容

將目前的排版規則：
```
禁止格式
- 不能用 Markdown 符號（星號、井號、橫線、反引號）
- 不能用條列式（「1. 2. 3.」或「•」）
```

修改為：
```
禁止格式
- 不能用 Markdown 標題符號（# ## ###）和粗體符號（**）
- 不能用傳統數字條列（1. 2. 3.）
- 【允許】使用 Emoji (✨/👉/🔮) 作為清單開頭，這在 Threads 很常見
```

### 5.2 優點分析

| 優點 | 說明 |
|------|------|
| **符合平台慣例** | Threads 上的爆文確實常用 Emoji 條列，這是平台文化 |
| **提升知識型可讀性** | 知識型貼文用 Emoji 條列會更清晰易讀 |
| **修改成本極低** | 只需修改一段 Prompt 文字 |
| **立即見效** | 修改後所有新生成的內容都會套用 |

### 5.3 缺點分析

| 缺點 | 說明 | 影響程度 |
|------|------|----------|
| **可能過度使用** | AI 可能會在不需要條列的地方也用條列 | 中 |
| **風格一致性** | 不同類型貼文的排版風格可能變得不一致 | 低 |

### 5.4 建議結論

> **建議執行** ⭐⭐⭐⭐

理由：修改成本極低，且確實能提升知識型、整理型貼文的可讀性。但建議加入更細緻的規則：

**建議的修改版本**：
```
### 排版格式
- 禁止：Markdown 標題符號（# ## ###）、粗體符號（**）
- 禁止：傳統數字條列（1. 2. 3.）
- 允許：Emoji 條列（✨/👉/🔮），但僅限於「知識型」「整理型」貼文
- 故事型、閒聊型貼文應保持自然段落，不使用條列
```

## 六、總結與執行建議

### 6.1 優先順序建議

| 優先順序 | 建議項目 | 建議 | 預估工時 | 價值/成本比 |
|----------|----------|------|----------|-------------|
| 1 | 新增「診斷型」貼文類型 | ✅ 執行 | 30 分鐘 | ⭐⭐⭐⭐⭐ |
| 2 | 放寬排版規則 | ✅ 執行 | 10 分鐘 | ⭐⭐⭐⭐ |
| 3 | 戰報閉環資料欄位 | ⏸️ 延後 | 2-3 小時 | ⭐⭐⭐ |

### 6.2 執行建議

**立即執行（本次迭代）**：
1. 新增「診斷型」前端定義
2. 放寬排版規則（加入 Emoji 條列允許）

**延後執行（待驗證需求後）**：
1. 戰報閉環資料欄位擴充
   - 先觀察戰報功能使用率
   - 收集用戶對「類型分析」的需求反饋
   - 確認標籤來源的最佳實作方式

### 6.3 風險提醒

1. **診斷型內容品質**：新增後建議先內部測試幾篇，確認 AI 生成的診斷型內容品質符合預期
2. **排版規則測試**：修改後建議測試各類型貼文，確認 Emoji 條列不會被過度使用
