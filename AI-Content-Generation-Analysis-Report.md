# 幕創行銷 AI 內容生成系統分析報告

**報告日期**：2025 年 1 月 14 日  
**報告作者**：Manus AI  
**版本**：1.0

---

## 摘要

本報告針對幕創行銷 Threads AI 教練系統的內容生成功能進行深入分析，回應學員反饋的「AI 感比較重」問題。經過程式碼審查與架構分析，我們識別出三個核心問題：**開頭句式同質化**、**範例過度具體化**、以及**風格學習機制不足**。本報告將詳細說明現有系統的優點與缺點，並提出具體的優化方案。

---

## 第一章：現有系統架構概述

### 1.1 內容生成流程

幕創行銷的 AI 內容生成系統採用多層次的提示詞架構，主要包含以下組件：

| 組件名稱 | 功能說明 | 檔案位置 |
|---------|---------|---------|
| **開頭規則模組** | 定義 7 種高效開頭模式與 4 種低效模式 | `shared/opener-rules.ts` |
| **內容過濾器** | 去除 AI 痕跡、成語替換、髒話過濾 | `server/contentFilters.ts` |
| **爆款因子系統** | 從資料庫取得爆款數據輔助生成 | `server/db.ts` |
| **用戶風格學習** | 分析用戶上傳的範文提取風格特徵 | `server/routers.ts` |

### 1.2 開頭模式設計

系統基於 1,739 篇爆款貼文的統計分析，定義了以下高效開頭模式：

| 模式名稱 | 效果倍數 | 範例句式 |
|---------|---------|---------|
| 冒號斷言 | 2.8x | 「學習的真相：不是你不夠努力」 |
| 禁忌/警告詞 | 2.4x | 「千萬不要在還沒搞懂這件事之前就開始」 |
| 反直覺陳述 | 1.8x | 「不是你不夠努力，是方向錯了」 |
| 數字開頭 | 1.7x | 「3 個讓你學了還是不會的原因」 |
| 對話式「你」 | 1.56x | 「你是不是也這樣？」 |
| 時間點 | 1.5x | 「這週我終於想通了」 |
| 結果導向 | 1.4x | 「結果我發現，問題根本不在這裡」 |

## 第二章：現有系統優點

### 2.1 數據驅動的內容策略

系統的最大優勢在於**數據驅動**的設計理念。開頭規則模組並非憑空設計，而是基於真實爆款貼文的統計分析得出。這確保了生成內容在結構上符合 Threads 平台的演算法偏好與用戶閱讀習慣。

系統會根據內容類型（如觀點型、故事型、知識型）推薦不同的開頭模式，並透過加權隨機選擇機制避免每次都使用同一種模式。這種設計在理論上應該能產生多樣化的內容。

### 2.2 完善的去 AI 化過濾系統

`contentFilters.ts` 實現了一套漸進式的去 AI 化過濾器，包含：

**成語殺手**：將 AI 常用的成語轉換為口語化表達。例如：
- 「盆滿缽滿」→「賺翻了」「賺到爆」「口袋滿滿」
- 「心力交瘁」→「累到不行」「身心俱疲」「整個人被掏空」

**廢話刪除**：移除 AI 常用的連接詞和冗詞，如「此外」「因此」「總而言之」「身為一個」等。

**情緒暴衝**：將陳述句轉換為更有情緒的表達，如「很多人」→「超多人」「一堆人」。

**髒話過濾**：使用多樣化替換詞避免重複，並優先使用用戶自己的情緒詞彙。

### 2.3 用戶風格學習機制

系統支援用戶上傳爆款貼文作為範文，並透過 LLM 分析提取風格特徵，包括：
- 語氣風格（如溫暖真誠、犀利直接）
- 常用句式特徵
- 口頭禪
- Hook 風格偏好
- 比喻風格
- 情緒節奏

這些特徵會在生成內容時作為參考，理論上能讓生成內容更貼近用戶的個人風格。

### 2.4 多層次的 IP 地基整合

系統會將用戶在「IP 地基」中設定的資料（職業、說話風格、人設三支柱、英雄旅程故事等）注入到生成提示詞中，確保生成內容與用戶的個人品牌定位一致。

## 第三章：現有系統缺點

### 3.1 開頭句式範例過於具體（核心問題）

這是造成「AI 感比較重」的最主要原因。在 `opener-rules.ts` 和 `routers.ts` 中，我們給 AI 的範例過於具體：

```typescript
// opener-rules.ts 中的範例
examples: [
  "你是不是也這樣？",
  "你有沒有發現",
  "你是不是總覺得時間不夠用",
  "你是不是也在等一個答案",
]
```

```typescript
// routers.ts 中的 Hook 策略說明
1. 鏡像策略：「你是不是也...」「有沒有人也會這樣...」
2. 反差策略：「我以為...但其實...」「我不是...但我還是...」
3. 解法策略：「...有問題？這幾點先看懂」
```

**問題分析**：LLM 在生成內容時會「學習」這些範例的結構，導致生成的內容開頭高度相似。即使我們在提示詞中明確禁止「直接複製範例」，LLM 仍然會傾向於使用相似的句式結構，因為這些範例在提示詞中佔據了顯著的位置。

### 3.2 Hook 策略說明重複出現

在 `routers.ts` 中，相同的三大策略說明在多個位置重複出現：

| 出現位置 | 行數範圍 | 內容 |
|---------|---------|------|
| generateHooks 函數 | 1111-1118 | Hook 三大策略說明 |
| hookStyleGuide 物件 | 1604-1611 | 各風格的範例句式 |
| generateDraft 函數 | 2725-2727 | 三大 Hook 策略 |
| generateQuickPost 函數 | 2985-2987 | 相同的策略說明 |
| optimizeDraft 函數 | 3567-3569 | 維度一：Hook 鉤子強度 |

這種重複強化了特定句式在 LLM「記憶」中的權重，導致生成內容更容易出現這些句式。

### 3.3 Few-Shot 範例可能不夠多樣

雖然系統會從爆款貼文資料庫中取得範例，但存在以下問題：

1. **範例數量有限**：每次只取 3-5 個範例，如果這些範例本身風格相似，AI 也會學到相似的表達方式。

2. **範例選擇邏輯**：目前是根據關鍵詞匹配選擇範例，可能導致選出的範例在主題上相似，但在風格上缺乏多樣性。

3. **用戶範文依賴**：如果用戶上傳的範文本身就有固定的開頭模式，系統會學習並複製這種模式。

### 3.4 風格學習機制的局限性

目前的風格學習機制存在以下問題：

1. **特徵提取過於表面**：系統主要提取「常用句式」「口頭禪」等表面特徵，而非深層的「說話邏輯」和「思維方式」。

2. **範文數量不足**：系統建議用戶上傳 3-5 篇範文，但這個數量可能不足以讓 AI 學習到用戶的完整風格。

3. **風格應用過於直接**：系統會直接在提示詞中列出用戶的常用句式，導致 AI 傾向於直接套用這些句式，而非學習其「精神」。

### 3.5 內容類型提示詞的固定結構

每種內容類型（如診斷型、反差型）都有固定的結構要求：

```typescript
// 診斷型貼文的結構要求
結構要求（嚴格遵守）：
1. 特徵召喚：開頭用「如果你經常...」「你有沒有這種經驗...」
2. 標籤揭曉：「那你可能是...」
3. 簡單解析：為什麼會有這種特徵
4. CTA：「你是哪一型？」「有沒有中？」
```

這種固定結構雖然確保了內容的完整性，但也限制了表達的多樣性。

## 第四章：問題根源分析

### 4.1 LLM 的「模式匹配」傾向

LLM 在生成內容時，會傾向於「模式匹配」——即找到提示詞中最明確的模式並加以複製。當我們在提示詞中提供具體的範例句式時，LLM 會將這些句式視為「正確答案」的模板，而非僅僅是參考。

### 4.2 「禁止」指令的有限效果

雖然我們在提示詞中加入了多處「禁止」指令：

> 「禁止直接複製上述範例的內容」  
> 「絕對禁止直接使用範文中的開頭句式」  
> 「每篇文章都要有全新的開頭，不能重複」

但 LLM 對「禁止」指令的遵從度有限，尤其當「正面範例」的權重過高時，「禁止」指令往往會被忽略。

### 4.3 風格學習的「過擬合」問題

當用戶上傳的範文數量有限時，系統容易出現「過擬合」——即過度學習範文中的特定句式，而非學習其整體風格。這類似於機器學習中的過擬合問題。

## 第五章：優化建議

### 5.1 短期優化（1-2 週可完成）

#### 5.1.1 移除或抽象化具體句式範例

**現狀**：
```typescript
examples: [
  "你是不是也這樣？",
  "你有沒有發現",
]
```

**建議改為**：
```typescript
styleDescription: "直接跟讀者對話，用第二人稱「你」開頭，製造共鳴感",
structureHint: "[第二人稱代詞] + [讀者可能的狀態/行為/感受]",
avoidPatterns: ["你是不是也", "你有沒有發現"] // 明確列出要避免的句式
```

#### 5.1.2 增加開頭模式的多樣性

目前系統有 7 種高效開頭模式，建議擴充至 15-20 種，並加入更多「非典型」的開頭方式：

| 新增模式 | 說明 | 結構提示 |
|---------|------|---------|
| 場景描寫 | 用具體畫面開場 | [時間/地點] + [正在發生的事] |
| 內心獨白 | 展現真實想法 | [情緒詞] + [內心的聲音] |
| 反問自己 | 自我對話 | [我問自己] + [問題] |
| 直接結論 | 先說答案 | [結論] + 句號（不是問號） |
| 數據震撼 | 用數據開場 | [具體數字] + [讓人意外的事實] |

#### 5.1.3 實作「反 AI 感」檢測

在內容生成後，加入一個檢測步驟，識別並標記常見的 AI 句式：

```typescript
const AI_PATTERNS = [
  /^你是不是(也|都|總是)/,
  /^你有沒有(發現|想過|經歷過)/,
  /^很多人(以為|覺得|認為).*但其實/,
  /^如果你(正在|經常|總是)/,
];

function detectAIPatterns(content: string): string[] {
  const firstLine = content.split('\n')[0];
  return AI_PATTERNS
    .filter(pattern => pattern.test(firstLine))
    .map(pattern => pattern.toString());
}
```

### 5.2 中期優化（1-2 個月）

#### 5.2.1 強化用戶風格學習機制

**增加範文數量要求**：建議用戶上傳至少 10 篇範文，並區分「爆款」和「一般」貼文，讓系統學習用戶的完整風格光譜。

**深度風格分析**：不只提取表面特徵，還要分析：
- 句子長度分布
- 換行頻率
- 情緒起伏節奏
- 論述邏輯結構
- 比喻和類比的使用方式

**風格權重調整**：讓用戶可以調整「風格相似度」，從「輕度參考」到「高度模仿」。

#### 5.2.2 實作「開頭生成器」獨立模組

將開頭生成從主要內容生成中分離出來，專門訓練一個「開頭生成器」：

1. **輸入**：主題、內容類型、用戶風格
2. **輸出**：5 個不同風格的開頭選項
3. **用戶選擇**：讓用戶選擇最喜歡的開頭
4. **學習回饋**：記錄用戶的選擇，優化未來的開頭建議

#### 5.2.3 建立「開頭多樣性」評分機制

追蹤每位用戶過去 30 天生成的內容，計算開頭的多樣性分數：

```typescript
interface DiversityScore {
  uniqueOpeningPatterns: number;  // 使用了多少種不同的開頭模式
  repetitionRate: number;         // 重複使用相同句式的比率
  averageSimilarity: number;      // 開頭之間的平均相似度
}
```

當多樣性分數過低時，系統應主動提醒用戶，並強制使用不同的開頭模式。

### 5.3 長期優化（3-6 個月）

#### 5.3.1 建立用戶專屬的微調模型

收集每位用戶的歷史生成內容和修改記錄，建立用戶專屬的風格模型。這需要：
- 足夠的數據量（至少 100 篇內容）
- 用戶的修改記錄（原始生成 vs 最終發布版本）
- 貼文表現數據（互動率、觸及率）

#### 5.3.2 實作「風格遷移」功能

讓用戶可以參考其他創作者的風格（經授權），系統分析目標風格的特徵，並將其應用到用戶的內容中，同時保留用戶的核心 IP 特色。

#### 5.3.3 AI 與人類協作的混合模式

重新定位 AI 的角色，從「內容生成器」轉變為「內容助手」：

| 階段 | AI 角色 | 人類角色 |
|-----|--------|---------|
| 靈感發想 | 提供主題建議、角度分析 | 選擇方向 |
| 開頭創作 | 提供 5 個開頭選項 | 選擇或修改 |
| 內容擴展 | 根據選定開頭生成內容 | 調整、補充 |
| 最終潤飾 | 檢查格式、建議優化 | 最終確認 |

## 第七章：結論

幕創行銷的 AI 內容生成系統在架構設計上具有堅實的基礎，包括數據驅動的開頭規則、完善的去 AI 化過濾器、以及用戶風格學習機制。然而，「AI 感比較重」的問題主要源於**提示詞中具體範例的過度使用**，導致 LLM 傾向於複製這些範例的結構。

解決這個問題的核心策略是：**從「給範例」轉向「給原則」**。與其告訴 AI「你可以這樣寫」，不如告訴 AI「你要達到什麼效果」和「你不能這樣寫」。這種方式能給予 AI 更大的創作空間，同時避免生成內容的同質化。

短期內，建議優先實施 P0 級別的優化（移除具體句式範例、增加避免句式清單），這些改動的實施成本低，但預期效果顯著。中長期則應著重於強化用戶風格學習機制和建立開頭多樣性評分系統，從根本上解決內容同質化的問題。
